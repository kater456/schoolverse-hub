
## Root causes

**iPhone (and some other) users can't load the site**
The app currently registers **two service workers on the same `/` scope**:
1. The Workbox SW generated by `vite-plugin-pwa` (`/sw.js`) — does aggressive `NetworkFirst` caching of every Supabase response.
2. A second SW at `/push-sw.js` registered manually by `src/lib/push.ts`.

iOS Safari (especially when the site has been added to Home Screen) fails silently or shows a white screen when two SWs fight for the same scope, and the broad `https://*.supabase.co/*` cache rule can serve stale auth/HTML to returning visitors. This matches the "iPhone can't access" symptom worldwide.

**"No device is receiving" push notifications**
Because the Workbox SW wins the registration race, the browser's single push subscription gets attached to **`/sw.js`** — which has **no `push` event handler**. So `send-push` happily delivers to the endpoint and the browser receives it, but nothing is shown to the user. From the admin's view, the dashboard correctly shows "0 delivered" for stale endpoints (or shows non-zero delivered but no notification ever appears on the device).

The database/RLS side is healthy: `push_subscriptions` allows public inserts, and the `send-push` function targeting logic is correct (anon devices with `school_id = null` are included in broadcasts).

## Fix

Consolidate to **one** service worker that owns both caching and push. Choice: drop the Workbox SW (we never asked for offline; only home-screen install matters) and keep `push-sw.js` as the single SW. Ship a one-release kill-switch worker at `/sw.js` so already-installed Workbox registrations on iPhones/laptops unregister themselves and free the scope.

### Changes

1. **`vite.config.ts`** — remove `VitePWA` plugin entirely. Keep the manifest as a static `public/manifest.webmanifest` (already linked from `index.html`). No more Workbox/runtime caching.

2. **`public/sw.js`** (new) — kill-switch worker per the PWA skill. On activate it deletes only Workbox-owned caches (so the FCM/push caches are untouched), claims clients, reloads open tabs, and `unregister()`s itself in `finally`. This evicts the stale Workbox SW from every returning device.

3. **`public/manifest.webmanifest`** (new) — port the manifest fields from `vite.config.ts` (`name`, `short_name`, `theme_color`, icons, `display: standalone`, `start_url: /`). `index.html` already references it.

4. **`src/lib/push.ts`** — before registering `/push-sw.js`, proactively `unregister()` any existing `/sw.js` registration (belt-and-braces in case the kill-switch hasn't run yet) and any existing push subscription tied to it, then re-subscribe through `/push-sw.js`. This guarantees the subscription endpoint is attached to the SW that actually has the `push` handler. Also ensures `push-sw.js` is registered with `updateViaCache: 'none'` and called once on every page load when permission is `granted`.

5. **`src/components/PushPrompt.tsx`** — keep behavior; just make sure it triggers `ensurePushRegistered()` immediately after permission grant so the new subscription is created against `/push-sw.js`.

6. **No database migration needed.** RLS, GRANTs, and the `send-push` function are already correct. We will run one **data fix** to clear out stale `push_subscriptions` rows whose endpoints were created against the old Workbox SW — the next page load will recreate them against `/push-sw.js` automatically.

7. **`supabase/functions/send-push/index.ts`** — add a lightweight `mode: "test"` branch so admins can hit "Send test to my own device" from the dashboard and verify delivery end-to-end. (Targeting logic untouched.)

8. **`src/components/PushBroadcastPanel.tsx`** — add a small **"Send test to myself"** button beside the main Send button, so you can verify any device is wired correctly without broadcasting.

### Why this fixes both problems

- One SW on `/` scope → iOS Safari stops choking → site loads for iPhone users worldwide, including those who had the app installed.
- The single SW (`push-sw.js`) is the one that owns both the push subscription and the `push`/`notificationclick` handlers → every device that previously "enabled notifications" will receive and display them again after one page visit.
- No aggressive Supabase cache layer → no more stale auth tokens or stale HTML on slow networks.

### Out of scope (intentional)

- We are **not** adding offline support; the user never asked for it and it's the source of these bugs.
- We are **not** changing CSP, Vercel config, or any RLS policies.
- We are **not** touching unrelated features (Reels, vendor profile, chat, AI, etc.).

### Verification

- Build, then load the published URL on:
  - iPhone Safari (fresh and previously-installed PWA) — page should render.
  - Desktop Chrome — Application tab should show only `/push-sw.js` active; `/sw.js` should be gone after one reload.
- Admin → Notifications → click **Send test to myself** → notification banner appears within a few seconds on the same device.
- Admin → Notifications → broadcast to "Everyone" → `sent` count in the response matches `select count(*) from push_subscriptions`.
